Overview

SafeNet is a utility designed to secure Network Messaages and ensure data integrity.

SafeNet is a "middleman" library, if SafeNet is active, then it's going to insert itself between network calls. This means that you should only have one implementation of SafeNet active, SafeNet will NOT load twice, so your executions should not be in this file.

Network messages are built upon using Blocks which are just fields in a table with a fancy name. Depending on the importance of your SafeNet message, you might want to use different, or multiple blocks of security. All of the blocks of security are interchangable with one another.

Validation

The first line of defense for SafeNet is a basic Server Client key pair. The client will periodically be sent a key with a seed you specify. The client will append all Network Messages registered in SafeNet with a key. If they cannot provide a valid key, their net message will be rejected.

Wormholes

Wormholes are a concept employed in SafeNet that allow you to basically open a mock 'port' to the client. If you can't reliably secure your network message, you should at least be able to reliably secure when it should be able to execute. Wormholes can be opened and closed at will. The Client cannot interact with wormholes. The Client will be sent a unique key when a SafeNet.Wormhole is opened to them, generated by your SafeNetSeed. They MUST reply with this key, or their net message will be rejected.

Type Parameters

It typically isn't enough for network messages to be EXPLICITLY typed as they are. People will still input invalid values, or do silly stuff. Developers will also occasionally screw up parameters too. You can add a "Network Parameter" to a SafeNet object, which will allow you to have strict, exclusive parameters, that SafeNet can look at.

Remember, never add the Player who is sending the message as a parameter.

SafeNet, like the Networking Library, requires your Type Parameters to be defined in the same order they are written.

Execution Rules

SafeNet allows you to manually whitelist groups, SteamIDs, or another variable, that must exist within the parameters provided. You can pass SafeNet a function when adding an ExecutionRule.

Exploit catching

SafeNet will automatically seek out common mistakes and exploits, and provided NetExploitsKillNetworkMessage is set to true, will kill the network message.

Sequential Execution

SafeNet is set up in a way similar to the network library, where each execution is ordered.

SafeNet UI

The SafeNet UI is designed to show you what and where, and when network messages are executed. The SafeNet UI will also notify you if any Network messages have been rejected; and will rat on whoever failed it, and why. From the SafeNet UI you can also configure a variety of options, such as auto-kick, auto-ban, which messages halt what, and so on. Of course, those variables can also be set by the Developer, but the SafeNet UI is considered as the final Arbiter of functionality.
