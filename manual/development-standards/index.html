Development Standards

The Development Standards of Werwolf Gaming servers is summed up by "the best that you can possibly do". We recognize that not everyone will be able to achieve an easy result.

Code that is at COMMIT REFERENCE HERE or September 2021 may not follow these conventions, if you see code that does not follow it: You can convert it to these conventions as long as it doesn't break anything. It isn't required, but would be very appreciated.

Objects

Actual Objects

Helix comes with the middleclass library, for objects that need to be instanced, we recommend using this library, Lua isn't really a good "Object Oriented" language, with the closest thing we have being MetaTables. The Middleclass library makes creating objects and instances of objects easy, and has inheritance.

See: {@middle-classes|Middle Classes}

Panel Objects

Panels should be Panel CLASSES, the only reason you should offshoot or dynamically extend (that is, to alter static variables or functions of the panel at runtime) and not extend by class, is if it's one line or very simplistic. Don't call DEFINE_BASECLASS and do a BaseClass function when not working with a Panel object, it looks ugly.

Non-purpose built panels (ie. buttons) should be able to function individually(i can open a dpanel and do panel:Add(class)) unless they have functionality that needs to be directly tied to the panel.

The Stack

Avoid unnecessary scope pyramids, if you're 6 scopes deep and all you're doing is Derma Panels, you're are most likely doing it wrong.

Conditionals

Conditionals should catch all circumstances they possibly can, things shouldn't ever be left to chance. Conditionals should be positive bias, meaning that most if statements should follow an if (x) then philosophy and not a if not (x) then philosophy, though this is difficult to follow, and isn't required.

Pyramids

Scoping is difficult, deciding where things should go for the best possible recognition is difficult. Scope pyramids should be limited to 5 scopes deep or less.

Function Pyramids

Functions should not be declared in functions. Excepting local functions.

Prototyping

Until code is committed to master, ALL CODE unless otherwise specified is to be considered prototype code, and is not production ready. Do not re-use other people's code that is prototyped.
